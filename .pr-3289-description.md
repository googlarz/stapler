## Thinking Path

> - Paperclip orchestrates AI agents for zero-human companies
> - Agents run in short heartbeat windows and lose all in-memory context between runs
> - When an agent learns something durable — a user preference, a decision, an environment quirk — there is today no place to store it beyond the issue it happened to be working on
> - This forces agents to either rediscover the same facts every run (wastes budget) or cram them into issue comments that don't belong there
> - This PR adds a per-agent, keyword-searchable memory store that survives across runs, exposed via a new HTTP API every adapter can already reach via `PAPERCLIP_API_KEY`
> - The benefit is that agents can accumulate knowledge across heartbeats without polluting issue threads or re-spending tokens on rediscovery

## What Changed

- New `agent_memories` table (migration `0055_agent_memories.sql`) with composite unique index `(agent_id, content_hash)` for dedupe and a `gin_trgm_ops` index for keyword search. Uses the `pg_trgm` extension that `0051_young_korg.sql` already installed — no new extensions.
- New Drizzle schema `packages/db/src/schema/agent_memories.ts` and shared `AgentMemory` / `AgentMemorySearchResult` / `AgentMemorySaveResult` types + zod validators.
- New `agentMemoryService(db)` with `save`, `search`, `list`, `getById`, `remove(id, agentId)`, `countForAgent`. `save` runs inside a single transaction: insert-or-skip via `ON CONFLICT DO NOTHING`, then count + prune oldest if over `PAPERCLIP_MEMORY_MAX_PER_AGENT` (default 500). `remove` is scoped by both `id` and `agentId` so it cannot delete another agent's row even if the caller skips the route pre-check.
- New HTTP routes under `/agents/:agentId/memories` — `POST`, `GET list/search`, `GET :id`, `DELETE :id`. Every handler gates on `assertCompanyAccess` first, then `assertAgentIdentity`.
- New `assertAgentIdentity(req, agentId)` helper in `server/src/routes/authz.ts`. If the actor is an agent key, it requires `req.actor.agentId === agentId`. Board actors fall through. This closes the `paperclipApiRequest` raw-URL escape hatch — the auth middleware at `server/src/middleware/auth.ts:162` resolves `req.actor.agentId` from the token's DB row (not the URL), so an agent X token hitting `/agents/OTHER/memories` is rejected at the guard.
- Activity log: emits `memory.saved` (only on genuine inserts, not dedupes, to avoid flooding) and `memory.deleted`.
- 24 new integration tests in `agent-memories-service.test.ts`, running against real embedded-postgres via the skip-aware `getEmbeddedPostgresTestSupport` harness.

## Why this approach

The original design called for `pgvector` + Ollama embeddings + per-adapter tool injection. Codex adversarial review killed two load-bearing assumptions:

1. **`embedded-postgres@18.1.0-beta.16` does NOT ship `pgvector`.** Inspection of the shipped extension directory (`node_modules/.pnpm/@embedded-postgres+darwin-arm64@.../native/share/postgresql/extension`) found no `vector.control`. `CREATE EXTENSION vector` would fail in the default dev path.
2. **"Zero adapter changes" was false.** Gemini / Codex / Cursor adapters inject `PAPERCLIP_API_KEY` but do not wire an MCP server. Only adapters whose underlying CLI natively discovers MCP (Claude Desktop, OpenCode) get MCP tools.

Pivoted to: ship keyword search via `pg_trgm` now (which IS already installed and in use for issue/comment search), with the HTTP API as the uniform surface every adapter can reach. Semantic / embedding-backed search becomes a follow-up gated on proving pgvector availability across deployment modes. The MCP tool layer ships in the next PR as a bonus for adapters that support it.

## Risks

- **`pg_trgm` relevance quality**: worse than semantic search for paraphrased queries. Agents are coached in `skills/paperclip/SKILL.md` (follow-up PR) to save content with explicit keywords. Future PR adds `pgvector`-backed semantic search once deployment availability is proven.
- **Schema hot path**: the new table adds 4 indexes (B-tree on `(agent_id, created_at)`, B-tree on `company_id`, GIN trgm on `content`, unique on `(agent_id, content_hash)`). Insert cost is higher than a bare table but still O(log n); for a cap of 500 rows/agent this is negligible.
- **Migration numbering collision**: uses idx `55` — if another PR lands `0055_*` first, this rebases cleanly with a rename to the next free idx.
- **Known non-issue — per-agent isolation**: tested at the service layer (cross-agent `remove` returns null) AND at the route layer (`assertAgentIdentity`). Auth middleware guarantees `req.actor.agentId` comes from the token's DB row, not the URL, so the guard cannot be bypassed by URL manipulation.
- **Content-size**: hard cap 4 KB per memory × 500 per agent = 2 MB/agent. Env-overridable via `PAPERCLIP_MEMORY_MAX_CONTENT_BYTES` and `PAPERCLIP_MEMORY_MAX_PER_AGENT`.

- **Provider / model**: Anthropic Claude Opus 4.6 via Claude Code
- **Exact model ID**: `claude-opus-4-6` (1M context variant: `claude-opus-4-6[1m]`)
- **Context window**: 1,000,000 tokens
- **Capabilities used**: extended tool use (Read / Edit / Write / Bash / Grep / Glob), adversarial design review via OpenAI Codex rescue agent, multi-agent parallel exploration, plan-mode gating
